# Migration to Version 2

## Summary

Simplifies agent versioning and the per-agent boot command format.

**Changes:**
1. **Remove `version` from `role.md` frontmatter.** Agent-level version stamping is dropped. Only `lore-repo.md` carries the framework version — one identifier per repo, stamped at creation and updated by `/lr:update`.
2. **Per-agent boot command format.** The `.claude/commands/lr-<agent-name>-agent.md` files are simplified to a one-line delegation pattern that references `agent-boot.md` (single source of truth for boot and operating instructions).

## Scope

Apply to a single repo. The `/lr:update` process invokes this migration once per repo that needs it.

## Steps

All steps must be idempotent — safe to re-run. Do **not** stamp the new version in `lore-repo.md` within this migration; the update process handles that after all steps succeed.

### Step 1: Remove `version` from all `role.md` files

For each agent in `<lore-agent-repo>/agents/`:

1. Read `<lore-agent-repo>/agents/<agent-name>/role.md`.
2. Parse the YAML frontmatter.
3. If a `version` key exists, remove it. Preserve `description` and any other keys.
4. Keep the body unchanged.
5. Write the file back, preserving line endings and general formatting where possible.

**Idempotent:** if `version` is already absent, this is a no-op for that file.

### Step 2: Regenerate per-agent boot commands

For each agent in `<lore-agent-repo>/agents/`:

1. Compute the new boot command content — exactly this single line plus a trailing newline:

   ```
   Read `${CLAUDE_PLUGIN_ROOT}/docs/agent-boot.md` and boot as agent `<agent-name>`.
   ```

   Note: at framework version 2 this template used a sibling-path form (`lore-framework/docs/agent-boot.md`). That form works only when `lore-framework/` is cloned next to the domain. The correct form uses `${CLAUDE_PLUGIN_ROOT}`, which Claude Code resolves to the installed plugin path. Migration 5 regenerates any v2-form boot commands to this plugin-safe form.

2. Target file: `.claude/commands/lr-<agent-name>-agent.md` (relative to the domain directory, not the repo).

3. **If the target does not exist**, create it with the new content.

4. **If the target exists and matches the new content exactly**, skip (idempotent).

5. **If the target exists and matches one of the known v1 templates** (see below), overwrite with the new content.

6. **If the target exists but diverges from any known v1 template**, the file was manually edited. Follow the manual-edit handling described in `update.md` — compute a three-way merge (old template → new template onto on-disk content), present it to the user, and proceed based on their choice.

#### Known v1 templates

The v1 `register-repo.md` template evolved through two forms. Both should be detected as "clean" and safely overwritten.

**Variant A — inlined operating summary (earlier v1):**

```
Read the following files and load yourself as the <Agent Name> agent:

1. `<repo>/agents/<agent-name>/role.md` — your role and identity
2. `<repo>/agents/<agent-name>/lore-context.md` — your compacted working knowledge

After reading both files, confirm you are loaded as <Agent Name> and briefly state your role and what you know.

## Operating as a Lore Agent

- Your lore is in `<repo>/agents/<agent-name>/lore/` — search it before making assumptions about things from previous sessions
- Your workspace is `<repo>/agents/<agent-name>/workdir/`
- You have access to the entire domain directory — all sibling repos and data
- At session end, if the user triggers finalization: `/lr:reflect` then `/lr:merge` (or `/lr:finalize` for both)
- Do not finalize unless the user explicitly triggers it
- For full operating instructions, see the lr plugin (`/lr:boot`)
```

**Variant B — three-file transitional form (later v1):**

```
Read the following files and load yourself as the <Agent Name> agent:

1. `lore-framework/docs/agent-boot.md` — how to operate as a lore agent
2. `<repo>/agents/<agent-name>/role.md` — your role and identity
3. `<repo>/agents/<agent-name>/lore-context.md` — your compacted working knowledge

After reading all three files, confirm you are loaded as <Agent Name> and briefly state your role and what you know.
```

When matching these variants, substitute `<Agent Name>` with the title-cased agent name, `<agent-name>` with the kebab-case directory name, and `<repo>` (the v1 template placeholder — preserved here as-was in the frozen v1 templates above) with the actual lore agent repo's directory name. A file matches a variant if the substituted template equals the on-disk content exactly (modulo trailing whitespace).

## Notes

- All steps in this migration are idempotent — safe to re-run after a partial failure.
- Removing `version` from `role.md` is a schema change. Framework tooling at version 2+ no longer requires or stamps the field.
- Step 2 is the main operational reason to run this migration — it aligns existing boot commands with the new single-source-of-truth boot flow.
- Do not update `lore-repo.md` frontmatter in this doc. The `/lr:update` process stamps the repo version only after all migrations for that repo succeed.
