# Migration to Version 5

## Summary

Makes the framework fully Claude Code plugin compatible by removing leftover sibling-path assumptions from user-side `.claude/commands/` artifacts.

**Changes:**
1. **Regenerate per-agent boot commands** to use `${CLAUDE_PLUGIN_ROOT}/docs/agent-boot.md` (the Claude-Code-resolved plugin install path) instead of `lore-framework/docs/agent-boot.md` (the pre-plugin sibling-path form).
2. **Prompt removal of orphaned pre-plugin skill duplicates** in `.claude/commands/` — commands like `lr-check.md`, `lr-reflect.md`, etc. that predate the plugin transition and are now superseded by the plugin's `/lr:<name>` skills. The user decides per file.

## Scope

Apply per lore agent repo — the `/lr:update` process invokes this migration once per repo that needs it, and `version-check.md` invokes it once for the booting agent's repo at boot time. The file modifications happen under `.claude/commands/` in the **domain directory**, not inside the repo, but migration is still triggered per repo (the first repo to reach v5 applies it; subsequent repos re-run idempotently and find nothing to do).

## Steps

All steps must be idempotent — safe to re-run. Do **not** stamp the new version in `lore-repo.md` within this migration; the update process handles that after all steps succeed.

### Step 1: Regenerate per-agent boot commands

Scan `<domain>/.claude/commands/` for files matching `lr-*-agent.md`. For each one:

1. Read the file content.
2. Determine whether it is a per-agent boot command (content contains the phrase `boot as agent`). If not, skip — it may be a plugin skill with a name ending in `-agent` (handled in Step 2) or a user-authored command.
3. Extract the agent name from the filename: strip the `lr-` prefix and the `-agent.md` suffix.
4. Compute the new boot command content — exactly this single line plus a trailing newline:

   ```
   Read `${CLAUDE_PLUGIN_ROOT}/docs/agent-boot.md` and boot as agent `<agent-name>`.
   ```

5. **If the on-disk content matches the new content exactly**, skip (idempotent — already migrated).
6. **If the on-disk content matches the known v2–v4 template** (the "clean pre-v5" form below), overwrite with the new content.
7. **If the on-disk content diverges from both**, treat as manually edited. Follow the manual-edit handling described in `update.md` — present a three-way merge (old template → new template onto on-disk content) and proceed per the user's choice.

Record each regenerated file for the final report.

#### Known clean pre-v5 template

Exactly this single line (plus a trailing newline):

```
Read `lore-framework/docs/agent-boot.md` and boot as agent `<agent-name>`.
```

A file matches when the template equals the on-disk content exactly (modulo trailing whitespace), with `<agent-name>` substituted by the kebab-case name extracted from the filename.

### Step 2: Prompt removal of orphaned pre-plugin skill duplicates

Scan `<domain>/.claude/commands/` for files matching `lr-*.md` that do **not** match `lr-*-agent.md` AND whose content does **not** contain the phrase `boot as agent`. For each such file:

1. Extract the skill name from the filename: strip the `lr-` prefix and `.md` suffix.
2. Check whether `${CLAUDE_PLUGIN_ROOT}/skills/<skill-name>/SKILL.md` exists. If not, this is a user-authored command unrelated to the framework — **leave it alone**.
3. If the plugin skill exists, this is an orphaned pre-plugin duplicate. The plugin now provides `/lr:<skill-name>`, so the local command is redundant and typically references stale sibling paths (`lore-framework/docs/...`, `lore-framework/commands/...`).

   Present the file to the user:
   - Path
   - First line (purpose summary, usually present)
   - A note: "The plugin provides `/lr:<skill-name>` — this local command is a pre-plugin duplicate."
   - Offer: **Delete** / **Keep** / **Keep all remaining**

4. If the user chooses delete, remove the file. If keep, leave it. If keep-all-remaining, skip all remaining candidates in this step.

Record each deletion and each skip for the final report.

**Idempotent:** once deleted, the file is gone and not rediscovered. Once kept, re-running prompts again (the user may choose differently next time) — this is intentional, since the framework cannot tell whether the user's "keep" was for this release or permanent.

### Step 3: Report

After both steps complete, print a summary:

- **Regenerated:** list of `lr-<agent-name>-agent.md` files rewritten in Step 1 (or "none").
- **Merged manually:** list of files that needed three-way merge (or "none").
- **Deleted:** list of orphaned `lr-<skill-name>.md` files removed in Step 2 (or "none").
- **Kept:** list of orphaned candidates the user chose to keep (or "none").

## Dry-Run Mode

When invoked from `/lr:update --dry-run`, detect all the same cases but do not modify any files. Report each would-be change:

- `would regenerate: <path>`
- `would prompt (manual edits): <path>`
- `would prompt (orphan): <path>`

Do not prompt the user in dry-run mode.

## Notes

- All steps in this migration are idempotent — safe to re-run after a partial failure.
- Step 1 is the main plugin-compat fix. Step 2 is hygiene cleanup.
- `.claude/commands/` lives in the **domain directory**, shared across all lore agent repos. Running this migration against the first repo in the domain handles all the shared commands; subsequent repos find nothing to do.
- Do not update `lore-repo.md` frontmatter in this doc. The `/lr:update` process (and boot-time version-check) stamps the repo version only after all migrations for that repo succeed.
