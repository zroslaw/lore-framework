# Release Notes — Version 7

## Summary

Version 7 introduces **session summaries** — a third phase of finalization that writes a short, committable markdown record of each session into the host agent's repo. Summaries preserve session narrative (what happened, not just what was learned) and create a searchable history of framework and agent evolution over time.

**Migration required:** no — `sessions/` directories are created on demand by the new process. Existing repos need no migration.

## What's New

### `/lr:summarize` skill and `docs/summarize.md`

A new skill, symmetric with `/lr:reflect` and `/lr:merge`. Thin `skills/summarize/SKILL.md` points to the authoritative `docs/summarize.md`, consistent with the skill-doc pattern.

The process:

1. Generates a UUIDv4 for the session
2. Resolves host agent, attached participants, user identity, and timestamps
3. Lists artifacts (files created / modified / deleted during the session)
4. Composes a 3–7 paragraph narrative using a structured prompt (context → what happened → plot twists → where it landed → next steps)
5. Shows the full summary to the user for review
6. On approval, writes to `<lore-agent-repo>/agents/<host-agent>/sessions/<YYYY>/<MM>/<YYYY-MM-DD>-<short-uuid>.md`
7. Emits the session UUID in user-visible output

### Finalize now orchestrates three steps

`/lr:finalize` = reflect → merge → **summarize**. Summarize is session-wide (one file regardless of attached guests) and runs after merge so the narrative can reference lore changes just made.

Summarize is **additive and non-blocking**: its failure does not roll back reflect or merge.

### UUID correlation to the private JSONL

The committed summary carries a UUIDv4 in frontmatter. The summarize step also emits that UUID in the agent's user-visible output, which lands in the Claude Code session JSONL on the user's machine. Users who want to later restore or inspect the raw session can grep for the UUID:

```bash
grep -rl "<uuid>" ~/.claude/projects/
```

This correlation works entirely on the user's own machine. The private JSONL is never copied into the public summary repo.

### New directory convention

`<lore-agent-repo>/agents/<agent-name>/sessions/YYYY/MM/` — created on demand when the agent hosts a session that runs summarize. Year/month nesting prevents single-directory bloat as summaries accumulate.

Frontmatter schema includes `uuid`, `start`, `end`, `host_agent`, `host_repo`, `participants`, `username`, `full_name`, `topics`, `artifacts`, and `consulted`. See `docs/summarize.md` for the full field reference.

### Review gate and privacy

Summaries are potentially public (committed to the agent repo). Two layers of defence:

1. **Narrative guidance** in the prompt: public-audience aware, no secrets or sensitive specifics, ask the user if unsure.
2. **Mandatory review gate**: the user sees the composed summary and must approve before anything is written.

No automated scrubbing in this release — the framework relies on agent judgment plus user review.

### Consult handling

`/lr:consult` invocations remain ephemeral (no finalization for the consultant). The host session's summary records consults in a `consulted` frontmatter field and, if the exchange materially shaped the session, in an optional **Consultations** body section.

## What Hasn't Changed

- Reflect and merge behavior, including per-agent iteration with attached guests
- Lore topic format, `lore-context.md` budget, descriptor file schema
- `/lr:attach`, `/lr:consult`, `/lr:recall` mechanics
- `/lr:check` — no new checks in this release (validation of summary integrity is deferred)

## Impact

- **No action needed for existing repos** — `sessions/` is created on demand.
- **First summary per agent** introduces the agent-specific `topics` tag vocabulary. Subsequent summaries should reuse established tags to keep the tag space coherent.
- **Start time is best-effort in v1** — the model estimates from session memory, rounded to the nearest 5 minutes. Reliable start-time capture is a tracked improvement.
- **Cross-repo guests** — if a guest is attached from a different lore agent repo, the summary still lives only in the host's repo. Guest repos do not receive a pointer file in this release (tracked as an improvement).
