# Release Notes — Version 3

## Summary

Version 3 introduces explicit lore recall, formalizes a subagent-based search pattern, and adds automatic version reconciliation at agent boot. **No file changes are required on the user side** — this is a feature release, not a migration.

## What's New

### `/lr:recall` — user-triggered lore recall

A new skill that lets you ask the loaded agent to pull lore topics relevant to the current session directly into its working context.

```
/lr:recall                       # contextual recall using current session
/lr:recall versioning decisions  # targeted recall with a hint
```

The agent prepares a search brief, dispatches an `Explore` subagent to read relevant topics, and presents the synthesis to you before continuing the task. See `docs/recall.md`.

### Subagent-scan as the preferred lore search pattern

Agents now search their lore by dispatching an `Explore` subagent with a structured brief, rather than reading topics directly into the main context. The LLM's semantic understanding handles phrasing mismatches that grep would miss, and the synthesis stays compact. Direct tools (Glob/Grep/Read/`git log`) remain available for known-target lookups. See `docs/lore-search.md`.

### Automatic version check at boot

`agent-boot.md` now compares the booting agent's repo `version` against the installed framework `VERSION`. If they differ, an upgrade runs automatically — applying migrations, displaying release notes, and stamping the new version. Safety invariants apply: uncommitted changes block the auto-upgrade, and any failure results in degraded-mode boot rather than blocking. See `docs/version-check.md`.

### Two-track update flow

`/lr:update` and the boot-time version check now handle two kinds of version-bump artifacts:

- **`migrations/<N>.md`** — for versions that require physical changes to user-side files (file edits, schema changes). Executed by the update process.
- **`release-notes/<N>.md`** — for feature releases that don't require user-side changes (informational only). Displayed to the user.

A version may have a migration, release notes, or both. At least one must exist for any version bump.

## What Hasn't Changed

- No file edits required in your agent repo
- No changes to lore topics, `lore-context.md`, or `workdir/`
- All existing skills, commands, and conventions still work as before
- Your agents will pick up the new search/recall behavior on next boot

## Background

Why a subagent scan instead of vector search? At current lore sizes, an LLM reading topics directly produces higher-quality semantic matches than embedding-based retrieval — with no new infrastructure. A vector-DB-backed approach is parked for when lore size makes the subagent scan impractical; see `vector-db-search-parked.md` in the `lore-architect` lore for the deferred design.
