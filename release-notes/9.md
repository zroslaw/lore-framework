# Release Notes — Version 9

## Summary

Version 9 carries three changes:

1. A new **worktree convention** — top-level repos in a domain stay on their default branch; any non-default-branch work lives in a git worktree under `<domain>/.worktrees/<repo>/<slug>/`. Documented in `docs/worktrees.md`.
2. A new **`/lr:init`** command that writes a compact statement of the convention into a framework-managed section of the domain's `CLAUDE.md`, so every Claude Code session starting in the domain reads it automatically.
3. **Fully automated finalization** — the approval gate on composed summaries (summarize step 10) and the approval gate on commit+push (finalize phase 4) are both removed. `/lr:finalize` runs end-to-end without prompts; the host summary is displayed inline at the end of summarize.

**Migration required:** no. All changes are behavioral or additive. Existing repos and agent files are untouched.

## What's New

### Worktree convention (`docs/worktrees.md`)

A new framework doc captures the full convention:

- **Invariant**: top-level directories in `<domain>/` that are repo checkouts stay on their default branch (whatever name that branch carries in each repo — `main`, `master`, `trunk`, etc.).
- **Rule**: any non-default-branch work — writing features, bug fixes, inspection checkouts — happens in a git worktree at `<domain>/.worktrees/<repo>/<slug>/`.
- **Optional pattern**: agents that want to keep notes on inflight worktrees may use `<lore-agent-repo>/agents/<agent-name>/worktrees/<slug>.md`. Structure and lifecycle are agent-defined; the framework does not mandate, scaffold, or check this.

Git worktree commands are standard — no framework wrappers. PR/MR workflow, review conventions, CI interactions, and merge mechanics remain repo-specific and live in agent lore (or repo-specialist agents reached via `/lr:consult` / `/lr:attach`).

See `docs/worktrees.md`.

### `/lr:init` skill and `docs/init.md`

A new skill, symmetric with the rest of the framework, distributes the convention into the domain-level `CLAUDE.md`:

- Target: `<cwd>/CLAUDE.md`.
- Marker protocol: framework-managed content is delimited by `<!-- lr:init:start -->` and `<!-- lr:init:end -->`. Content outside the markers is never touched.
- Behavior: create the file if missing; prepend the managed section if the file exists without markers; no-op if the managed section matches the canonical payload; show-diff-and-confirm if the managed section has drifted.
- Idempotent: safe to run any number of times. Re-run after framework version bumps to refresh the managed section.

The v1 canonical payload is a compact statement of the worktree rule plus a link to the full convention doc. Future framework versions may extend the payload (domain intro, agent registry pointers, invocation tips) — all inside the same marker block; the mechanism stays the same.

See `docs/init.md`.

### Skill-doc alignment

`skills/init/SKILL.md` is a thin one-line pointer to `docs/init.md`, consistent with the skill-doc pattern used by the rest of the framework.

### Fully automated finalization

Two review gates in the finalization flow are removed:

- **Summary approval** (summarize.md step 10) — previously the host summary and all guest summaries were shown to the user for approval before any file was written. That step is gone; summarize writes directly. A new step 12 emits the UUID/paths **and displays the host summary contents inline** so the user sees what was recorded.
- **Commit+push approval** (finalize.md phase 4) — previously each repo's pending diff was shown with a "wait for approval" prompt before commit. That prompt is gone; each repo is staged (`agents/`), committed with `Finalize session <short-uuid>`, and pushed without user interaction.

Post-hoc review channel: git history. The user sees what was pushed after the fact and can amend or revert if something slipped through.

Privacy defense is now the narrative-guidance prompt in `summarize.md` alone — the agent is the sole filter at write time. Failure modes involving user rejection have been removed from the summarize failure table; a model that cannot produce a safe summary should report an error rather than writing something compromised.

See `docs/summarize.md` and `docs/finalize.md`.

## What Hasn't Changed

- Boot, reflect, merge — no behavioral changes.
- `/lr:check` — no new checks in this release. Invariant enforcement (warning when a top-level repo dir is on a non-default branch) is deferred until drift is observed in practice.
- `/lr:pull-domain` — continues to operate on top-level repos only; `.worktrees/` is untouched.
- Lore topic format, `lore-context.md` budget, descriptor file schemas.
- `/lr:attach`, `/lr:consult`, `/lr:recall` mechanics.
- Standalone `/lr:reflect`, `/lr:merge`, `/lr:summarize` still don't commit — only `/lr:finalize` commits and pushes.

## Impact

- **No action needed for existing repos.** No migration, no file regeneration, no version-stamp changes beyond the routine `8 → 9` stamp on next `/lr:update` or auto-boot upgrade.
- **Users wanting the convention surfaced in their domain** run `/lr:init` in the domain directory. The framework-managed section is prepended to existing `CLAUDE.md` content (if any) or becomes the whole file (if it didn't exist).
- **`/lr:finalize` no longer prompts.** Previously two confirmation gates (summary content, per-repo commit); now both are removed. Agent judgment is the sole privacy filter at write time; git history is the post-hoc review channel. Users who want the old review behavior can run `/lr:reflect`, `/lr:merge`, and `/lr:summarize` separately, then commit/push manually — standalone skills still don't commit.
- **Agent knowledge about repo state** remains tied to top-level (default-branch) checkouts — the invariant preserves the meaning of existing lore that references "this repo's code."
