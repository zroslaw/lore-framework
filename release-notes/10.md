# Release Notes — Version 10

## Summary

Version 10 introduces **`/lr:spawn-teammate` (BETA)** — a new skill that spawns one or more lore agents as Claude Code Agent Teams teammates. The current session becomes the team lead; each teammate is a separate, independent Claude Code instance booted as the specified lore agent via `agent-boot.md`. This is the framework's first integration with Agent Teams: explicitly beta, intentionally minimal.

**Migration required:** no. The change is additive — a new skill plus a new doc. No existing files are modified.

## What's New

### `/lr:spawn-teammate` skill (BETA)

A new plugin skill, symmetric with the rest of the framework:

- `skills/spawn-teammate/SKILL.md` — thin one-line pointer.
- `docs/spawn-teammate.md` — full procedure.

Usage:

```
/lr:spawn-teammate <agent-name>            # spawn one teammate
/lr:spawn-teammate <name-1> <name-2> ...   # spawn multiple at once
/lr:spawn-teammate                         # infer from session context (asks if ambiguous)
```

Mechanism in four moves:

1. **Enumerate** available lore agents in the domain (every subdir with `lore-repo.md` → every `agents/*/role.md`).
2. **Resolve** user input — explicit args, context inference, or fuzzy/typo tolerance — into a concrete agent set, asking when ambiguous.
3. **Compose** a teammate spec per resolved agent: a kebab-case name and a verbatim spawn prompt of the form `Read ${CLAUDE_PLUGIN_ROOT}/docs/agent-boot.md and boot as agent <name> from <abs-path>. After boot, await further instructions from the team lead via SendMessage.`
4. **Invoke** Agent Teams via a single natural-language directive that creates or extends the team using the composed specs verbatim.

The skill does not introduce any new file format or per-agent metadata — it is a thin name-resolution and spawn-prompt-composition layer over Agent Teams' natural-language interface.

### Prerequisites

`/lr:spawn-teammate` requires Claude Code's experimental Agent Teams to be enabled. Either:

- Set the environment variable: `export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`, or
- Add to `~/.claude/settings.json`:
  ```json
  {
    "env": {
      "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
    }
  }
  ```

Agent Teams requires Claude Code v2.1.32 or later. See https://code.claude.com/docs/en/agent-teams for the official Agent Teams documentation.

### BETA caveats

The skill is explicitly marked beta in the description, the doc, and the first-time runtime output. Reasons:

- **Agent Teams itself is experimental** — its own contract may evolve.
- **Lore writes are last-write-wins.** When two teammates running as the same lore agent (or one teammate plus the lead) write concurrently, no serialization is enforced. Mitigation: defer lore-changing work to finalization.
- **Finalization across teammates is not yet automated.** Each teammate runs its own `/lr:finalize` before disbanding; the lead's session is finalized separately.
- **Agent Teams' own limitations apply:** one team per session, no session resumption with in-process teammates, no nested teams, lead is fixed for the team's lifetime.

The skill name and high-level behavior (resolve names → spawn teammates booted as the named agents) are stable for the duration of the beta. Internal procedure, presentation, and beta caveats may change as real usage surfaces what works.

### How `/lr:spawn-teammate` differs from `/lr:attach`, `/lr:consult`, and `/lr:recall`

The existing trio operates within a single Claude Code session: `/lr:recall` searches the lore of agents already loaded; `/lr:consult` dispatches a focused subagent for a one-shot question to an unloaded agent; `/lr:attach` loads another agent as a guest into the host session for sustained co-work.

`/lr:spawn-teammate` is fundamentally different: it spawns separate, independent Claude Code instances. The lifecycle, communication channel (Agent Teams mailbox), and finalization story all differ. The trio (in-session) and spawn-teammate (across sessions) coexist and compose — there is no overlap to consolidate.

## What Hasn't Changed

- All existing skills and behaviors are unchanged. No backward-incompatible changes.
- No file edits required in any agent repo.
- `/lr:check` — no new checks. (Validators for spawn-teammate-related state are deferred until usage shows what to check.)
- Boot flow, lore topic format, descriptor file schemas, version-stamp mechanics.

## Impact

- **No action needed for existing repos.** No migration; no file regeneration. Repos stamped at v9 will auto-stamp to v10 on the next boot via the routine version-check flow.
- **Users wanting to try `/lr:spawn-teammate`** must enable Agent Teams (env var or settings.json), then invoke the skill from a domain directory containing at least one lore agent repo.
- **`CLAUDE.md` reaches teammates.** Each teammate is a regular Claude Code session that loads the working-directory `CLAUDE.md` automatically. The framework-managed section written by `/lr:init` (worktree convention etc.) propagates to teammates without extra work.
