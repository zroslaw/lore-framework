# Release Notes — Version 8

## Summary

Version 8 restructures the finalization lifecycle. Reflect stays inline and host-first, but **merge moves to parallel subagents, one per active agent, each booted as its target agent**. Committing and pushing centralize into a new **Phase 4** of `/lr:finalize` — one commit per touched repo, review-gated. Guests with lore updates now get a **short guest summary** in their own repo (sharing the host's session UUID). A new **conflict-resolution procedure** handles push races from concurrent finalizations.

**Migration required:** no. All changes are behavioral; `sessions/` directories under guest agents are created on demand.

## What's New

### Merge in booted subagents, parallel

Merge no longer runs inline. `/lr:merge` now spawns one subagent per active agent (host + any attached guests). Each subagent boots as its target agent per `agent-boot.md` and then runs the merge procedure scoped to itself. All subagents launch in parallel. This is uniform for single- and multi-agent sessions — a single-agent session just spawns one subagent.

Booting each subagent as its target agent gives it the agent's role and lore-context naturally — the same pattern `/lr:consult` already uses.

See `docs/process-merge.md`.

### Commit and push centralized in finalize Phase 4

Merge no longer commits per agent. Summarize doesn't commit either. Both phases leave changes on disk; `/lr:finalize` Phase 4 produces exactly one commit per touched repo (host's lore + host summary + any guest lore + any guest summary), then pushes each, with a mandatory review gate per repo.

Standalone `/lr:reflect`, `/lr:merge`, and `/lr:summarize` do not commit or push — users commit themselves when running a phase alone.

See `docs/finalize.md` (new).

### Guest session summaries

When an attached guest had lore updates during merge, summarize now writes a **short guest summary** to that guest's repo under `agents/<guest>/sessions/YYYY/MM/`. The guest summary uses the same session UUID as the host summary and back-references the host's canonical narrative. Guests attached but with no lore updates get no summary. Consulted agents get no summary either — their involvement is recorded only in the host summary's `consulted` field.

Guest frontmatter is slimmer than host: `uuid`, `date`, `role: guest`, `host_agent`, `host_summary_repo`, `host_summary_path`, and a `lore_changes` list. The body is a short participation + contribution + bulleted lore-updates + back-reference.

See `docs/summarize.md` for the schema and process.

### Conflict resolution on push

When Phase 4 push is rejected due to conflicts inside an agent subtree (e.g., two users finalize the same agent concurrently), finalize triggers a new conflict-resolution procedure: one subagent per conflicted agent, booted as that agent, merges local + remote lore and retries push **up to 3 total attempts**. Conflicts outside agent subtrees, or role-redefinition conflicts, fall back to the user.

See `docs/resolve-conflicts.md` (new).

### New `docs/finalize.md` and skill-doc alignment

`skills/finalize/SKILL.md` is now a thin pointer, consistent with reflect/merge/summarize. All orchestration (the four phases, failure handling, cross-repo handling, invariants) lives in `docs/finalize.md`.

## What Hasn't Changed

- Reflect is still inline, host-first, per active agent. Reflect needs session context — which a fresh-booted subagent wouldn't have — so its per-agent iteration stays in the host session.
- Lore topic format, `lore-context.md` budget, descriptor file schema
- `/lr:attach`, `/lr:consult`, `/lr:recall` mechanics
- `/lr:check` — no new checks in this release. Validators for guest-summary orphans and host/guest UUID correlation are deferred.

## Impact

- **No action needed for existing repos.** No migration.
- **Parallel merge** shortens finalize wall-clock time for multi-agent sessions, at a small per-spawn overhead (~10–20k tokens for each subagent's role + lore-context read).
- **Cross-repo guests now receive a summary** in their own repo — if a guest is attached from a repo with different visibility than the host's, review the guest summary's contents with that destination repo in mind at the Phase 4 review gate.
- **Standalone skills no longer commit.** Users who relied on `/lr:merge` auto-committing need to commit manually (or switch to `/lr:finalize`).
- **Push can be partially asymmetric across repos.** If the host repo's push succeeds but a guest repo's push fails, the public record is momentarily split. The user can retry the failed repo's push manually; the local commit is intact.
