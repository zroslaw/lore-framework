# Release Notes — Version 4

## Summary

Version 4 introduces cross-agent collaboration: `/lr:attach` for sustained multi-agent sessions and `/lr:consult` for one-shot specialist queries. Recall and finalization become multi-agent-aware when guests are attached. **No file changes are required on the user side** — this is a feature release, not a migration.

## What's New

### `/lr:attach` — sustained multi-agent sessions

A new skill that loads another lore agent into the currently booted host session. The host remains the sole executor; the guest's role and lore-context join the host's working context so the host can work with the union of both agents' knowledge over many turns.

```
/lr:attach <agent-name>    # attach a guest
/lr:attach                 # list currently attached guests
```

Use attach when a task genuinely spans two (or more) domains and you'll be thinking in both across many turns. See `docs/attach.md`.

Key properties:
- **Host / guest** — one host per session (the originally booted agent), zero or more guests. Guests are knowledge loads, not additional executors.
- **Multi-guest** — attach more than once to compose multiple specialists.
- **Version reconcile** happens automatically for the guest repo in a subagent, so migration noise stays out of the host context.
- **Host-lore-wins** on conflicts — the guest's perspective informs but does not override.
- **No detach** in v4. Once attached, guests stay for the session and participate in finalization.

### `/lr:consult` — one-shot specialist queries

A new skill that asks an **unloaded** lore agent a focused question. A subagent boots the consultant, searches its lore, synthesizes a response with pointers to specific lore topics and workdir tools, and exits.

```
/lr:consult <agent-name>         # contextual — brief built from session
/lr:consult <agent-name> <hint>  # focused — hint layered on session context
```

Use consult when a focused question will get what you need without loading the agent. The consultant itself is unaffected — no reflection, no merge. After a consult, the host may read specific referenced files directly, but does not continue searching the consultant's lore (that's attach territory). If you realize you need sustained engagement, escalate with `/lr:attach <same-agent>`.

See `docs/consult.md`.

### Recall fans out across active agents

When guests are attached, `/lr:recall` automatically searches the lore of every active agent — host + all attached guests — by dispatching one `Explore` subagent per agent in parallel. Results are presented grouped by agent so the user can see whose lore each finding came from. When there are no guests, behavior is unchanged. See `docs/recall.md` and `docs/lore-search.md`.

### Finalization iterates per active agent

When guests are attached, `/lr:reflect`, `/lr:merge`, and `/lr:finalize` iterate per active agent, sequentially, in host-first order. Each iteration is scoped by that agent's own `role.md` + `lore-context.md` — so each agent learns only what fits its role. Topics legitimately relevant to multiple agents may appear in each agent's lore (duplication for shared knowledge is allowed and preferred over each agent re-learning the same thing separately). See `docs/process-reflection.md` and `docs/process-merge.md`.

### The trio in one line each

- **`/lr:recall [hint]`** — search lore of agents *already loaded* (host + attached guests). Fans out.
- **`/lr:consult <agent> [hint]`** — one-shot question to an *unloaded* agent. Subagent boots, answers, exits.
- **`/lr:attach <agent>`** — load another agent's full lore-context into the host session for sustained co-work.

## What Hasn't Changed

- No file edits required in your agent repo
- No changes to existing lore topics, `lore-context.md`, `role.md`, or `workdir/`
- Single-agent sessions (no attached guests) behave exactly as before — recall operates on the host, finalization runs once for the host
- All existing skills, commands, and conventions still work as before
- `/lr:check` is unchanged

## Background

Why introduce attach and consult when lore-search already exists? Recall works only within a single loaded agent's lore. That's correct for its role, but it leaves a gap: narrow utility domains (external API clients, per-repo commit conventions, specialized tooling) show up across many agents without being any one agent's primary role. Before v4, each agent had to re-learn such a domain in its own lore — duplication, divergence, and no compounding of learning.

Attach and consult give two complementary ways to share knowledge across agents without duplication:

- **Consult** — one-shot, lightweight; the consultant's lore is the single source of truth and other agents ask it questions.
- **Attach** — sustained, heavyweight; multiple agents' knowledge joins the host session for work that genuinely spans domains.

Recall extends naturally to this model by fanning out over whatever is loaded. Finalization iterates per active agent so each one's lore stays coherent with its own role.
