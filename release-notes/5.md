# Release Notes — Version 5

## Summary

Version 5 finishes the Claude Code plugin transition. Framework docs and generated artifacts now consistently use `${CLAUDE_PLUGIN_ROOT}` for plugin files (never a sibling-path `lore-framework/` assumption) and framework docs use a single consistent placeholder vocabulary for lore agent repos.

**Migration required:** yes — migration 5 regenerates per-agent boot commands and prompts to remove orphaned pre-plugin skill duplicates in `.claude/commands/`.

## What's New

### Plugin-compat fix for per-agent boot commands

Per-agent boot commands (`.claude/commands/lr-<agent-name>-agent.md`) are now generated with `${CLAUDE_PLUGIN_ROOT}/docs/agent-boot.md` instead of the pre-plugin `lore-framework/docs/agent-boot.md` sibling-path form. The old form only worked when the framework was cloned as a sibling directory of the domain; installations that rely solely on the plugin (e.g., via marketplace install) saw broken boots.

The v5 migration regenerates every `lr-<agent-name>-agent.md` in your domain's `.claude/commands/`:
- If the file matches the pre-v5 template exactly, it is silently rewritten.
- If the file has been manually edited, a three-way merge is presented for your decision.

### Cleanup of orphaned pre-plugin skill duplicates

Installations that predate the plugin transition carry files like `.claude/commands/lr-check.md`, `lr-reflect.md`, `lr-create-agent.md`, etc. These duplicate plugin skills (`/lr:check`, `/lr:reflect`, `/lr:create-agent`) and typically reference stale sibling paths.

The v5 migration detects each such orphan and prompts per file: **Delete / Keep / Keep all remaining**. User-authored commands unrelated to framework skills are left alone.

### New `/lr:check` detections

- **Orphaned pre-plugin commands** — flags any `lr-<skill-name>.md` in `.claude/commands/` whose `<skill-name>` matches a current plugin skill. These are redundant duplicates.
- **Legacy sibling-path per-agent boot commands** — flags any `lr-<agent-name>-agent.md` still using `lore-framework/docs/agent-boot.md`. Running `/lr:update` regenerates them to the plugin-safe form.

### Placeholder vocabulary consolidated

Framework docs previously used a zoo of overlapping placeholders (`<repo>`, `<repo-path>`, `<agent-repo>`, `<agent-repo-path>`, `<absolute-path-to-guest-repo>`) for the same concept. They are now a single consistent set:

- **`<lore-agent-repo>`** — path to a lore agent repo. Used in `/lr:check`, `/lr:update`, `/lr:boot`, `git -C` examples, and elsewhere.
- **`<guest-lore-agent-repo>`** — same, but specifically the guest in `/lr:attach`.
- **`<agent-name>`** — the kebab-case agent name.
- **`<domain>`** — the domain directory.
- **`${CLAUDE_PLUGIN_ROOT}`** — Claude Code's resolved plugin install path.

The `lore-` prefix on repo placeholders avoids confusion with unrelated repos that may live alongside lore agent repos in the same domain. See `docs/conventions.md` for the full vocabulary.

## What Hasn't Changed

- No changes to lore topics, `lore-context.md`, `role.md`, `lore/`, or `workdir/` contents
- Existing attach / consult / recall / finalization behavior is unchanged
- Host/guest model, per-agent iteration, and all v4 collaboration mechanics are unchanged
- Existing migration 2 behavior is preserved — only its template output form is updated for plugin-compat

## Background

Why ship this now? The framework moved to plugin distribution several versions back, but a few references to the pre-plugin sibling-path form (`lore-framework/docs/...`) lingered in two template spots: `docs/register-repo.md` and `migrations/2.md`. Both spots produce user-side artifacts in `.claude/commands/`, so the stale template propagated to every registered agent boot command on every install that had already moved through migration 2. This worked as long as users kept a local clone of `lore-framework/` alongside their domain — which was the default during the transition — but broke on any plugin-only install.

The placeholder consolidation is a smaller cleanup motivated by the same audit. The four legacy placeholder names all meant the same thing, making it easy for readers to assume they were distinct concepts and miss that they could confuse a lore agent repo with some other repo sitting in the domain. A single `<lore-agent-repo>` removes the ambiguity.
